## 1. Что это

Одностраничный статический дашборд (`index.html`) по сети из 18 пунктов выдачи Uzum.
Без бэкенда — весь расчёт в браузере. Три раздела (вкладки сверху):

- **Обзор** — KPI, симулятор роста заказов, кнопки «Безубыточность сети» / «150 заказов/день»
  / «Текущий результат», графики (Факт vs Безубыток / Прогноз / Яндекс-пик / P&L), таблица по точкам.
- **Недельная история** — заказы/день и объём/день по неделям: сводка по сети + таблица по ПВЗ.
- **Загрузить файл** (под админ-паролем) — загрузка нового `узум.xlsx`, пересчёт в браузере.

Вся бизнес-логика (комиссии по сегментам, гарантии, безубыток) — из оферты Uzum и отчётов.

## 3. Деплой (GitHub → Railway)

1. Положи файлы в репозиторий, запушь в GitHub.
2. Railway → New Project → Deploy from GitHub repo → выбери репозиторий.
3. Railway прочитает `package.json` и запустит `npm start`, который поднимет статику
   (`serve -s . -l $PORT`). `$PORT` Railway задаёт сам.
4. **Привязка домена `punktividachi.com` (Cloudflare → Railway):**
   - Railway → проект → **Settings → Networking → Custom Domain** → ввести `punktividachi.com`
     (и/или `www`). Railway покажет целевую запись — обычно **CNAME** на хост вида
     `…up.railway.app`.
   - В **Cloudflare → DNS → Records** добавить эту запись: тип **CNAME**, name `@` (или `www`),
     target — значение от Railway.
   - Для корня (`@`) Cloudflare применит CNAME-flattening автоматически.
   - Рекомендация: на первичной проверке поставить **Proxy status = DNS only** (серое облако),
     убедиться что сайт открылся, потом при желании включить оранжевое облако (проксирование/CDN).
   - Дождаться выдачи TLS-сертификата (Railway/Cloudflare сделают сами), затем открыть домен.
5. Любой `git push` в основную ветку → Railway пересобирает автоматически.

> Альтернатива статике: на Cloudflare Pages этот же `index.html` грузится напрямую
> (Workers & Pages → Pages → Upload assets), домен привязывается в один клик.

## 4. Модель данных

Внутри `index.html` два объекта:

- **`DATA`** — массив из 18 ПВЗ. Поля:
  - Справочные (из оферты/отчётов, меняются редко): `code, aka, city, seg (1/2/3),
    comm (0.06/0.05), opex_usd, uk (затраты на УК, $/мес ≈ 320), guarFull (сум/мес),
    fcMonth1 (прогноз 1-го мес, зак/день), yandex (пик, зак/день), gmvFc_usd, start (дата)`.
  - Из дневного файла (обновляются загрузкой): `w1..w4 (заказы/день по неделям),
    runRate (заказы/день последней полной недели — используется в расчётах Обзора),
    periodAvg, revW3, revPeriod, avgCheck`.
- **`WK`** — недельная история: `weeks[] (метки-диапазоны дат), dayCounts[] (дней в неделе),
  runIdx (индекс run-rate-недели), net{ord[],rev[]}, pvz{ code: {aka,city,seg,ord[4],rev[4]} }`.
  `ord` = заказы/день, `rev` = выручка/день.

**Что откуда:** заказы и выручка — ТОЛЬКО из `узум.xlsx`. Сегменты, комиссии, опекс,
гарантии, прогнозы, Яндекс-пик — из `Эталон_ПВЗ.xlsx → Справочник` (в дневном файле их нет).

Сегменты (из оферты, не менять без основания): Сег 1 = Ташкент (6%, гарантия 17,4 млн);
Сег 2 = Ташкентская область + Самарканд/Бухара/Андижан/… (5%, 14 млн); Сег 3 = прочие (5%, 11,4 млн).
Конкретно: 9 ташкентских = Сег 1; Самарканд(Рудаки, Андижанская), Бухара, оба Алмалыка,
Бука, Пскент, Газалкент = Сег 2; Маргилан = Сег 3.

## 7. Обновление данных

Два пути:

**А. Быстро, в браузере (без редеплоя):** вкладка «Загрузить файл» → админ-пароль →
выбрать `узум.xlsx`. Пересчёт мгновенный, но только в текущей сессии. Эталонные (вшитые)
данные не меняются — перезагрузка вернёт исходные. Удобно показать инвестору свежие цифры.

**Б. Навсегда (для всех зрителей) — через Claude Code, с НАКОПЛЕНИЕМ:**
```
python build_data.py узум.xlsx Эталон_ПВЗ.xlsx index.html
```
Что делает скрипт:
- **Накапливает** недели в `Эталон_ПВЗ.xlsx`: уже зафиксированные ПОЛНЫЕ недели (7 дней)
  НЕ переписываются; дописываются только новые недели справа; ПОСЛЕДНЯЯ неполная неделя
  обновляется при каждом прогоне (пока в неё дотекают дни) — как станет полной, фиксируется.
- Лист «Справочник» только читает (твои входы: сегмент/комиссия/опекс/УК/гарантия/прогнозы).
- Пишет в `data.json` ВСЮ накопленную историю недель (`DATA`/`WK`): страница «Недельная история»
  показывает все недели (с горизонтальной прокруткой), а «Обзор» использует последнюю полную
  неделю как run-rate. `FACT` (Сатори) ведёт `build_satori.py` — тоже в `data.json`.
- Если недельные листы эталона пустые — создаёт их с нуля (для свежего эталона достаточно Справочника).

Затем синхронизируй копию и закоммить:
```
copy index.html Uzum_дашборд.html
git add -A && git commit -m "data update" && git push
```
Railway пересоберёт автоматически.

> Важно: подавай в скрипт КУМУЛЯТИВНЫЙ `узум.xlsx` (со всей историей до текущей даты).
> Старые полные недели в эталоне всё равно не перезапишутся, а последняя неполная корректно
> доберёт дни. Тот же принцип — у загрузки через портал (показывает последние 4 недели файла).

**Безопасность загрузки (заложена в код, не ломать):** парсер проверяет колонки
(Адрес, ПВЗ, Дата, Кол-во заказов, Общая сумма) и листы-даты; при несовпадении формата —
отказ с понятной ошибкой, текущие данные не трогаются. При открытии дашборд делает снимок
эталона (`BACKUP`), есть кнопка «Вернуть эталонные данные».

## 8. Логика недель (для справки)

- Недели — календарные, Пн–Вс. Показываются последние 4 недели данных.
- День в неделе считается «с данными», если в этот день у точки есть числовое «Кол-во заказов».
- `заказы/день` = сумма заказов недели ÷ дни с данными. `выручка/день` — аналогично.
- `runRate` (для Обзора) = заказы/день последней ПОЛНОЙ недели (7 дней).
- Неполная неделя (<7 дней) помечается звёздочкой и штриховкой, в трендах не экстраполируется.

## 9. Что НЕ делать

- Не менять цвета/шрифты/тему, не добавлять светлые фоны или сторонние UI-киты.
- Не делать кнопки разноцветными — все `.bebtn` фиолетовые.
- Не подменять «средние на ПВЗ» суммами в итоговых строках приложений.
- Не убирать пометку неполных недель.
- Не хранить пароли/секреты в открытом виде, если переходишь на серверную защиту.
- Не ломать неразрушающую загрузку (снимок эталона + валидация + откат).
- При расширении DATA/WK — сохранять имена существующих полей (от них зависит весь рендер).

## 10. Быстрый старт для Claude Code

1. Прочитать этот файл.
2. Открыть `index.html`: `<style>` (дизайн-токены) и `<script>` (render-функции). Сами данные — в `data.json` (`DATA`/`WK`/`FACT`/`SAL`), страница грузит их через `<script src="/data.js">` (`server.js` отдаёт за паролем).

4. Проверка перед коммитом: открыть страницу в браузере (через настоящий `server.js` локально, т.к. данные за паролем); для синтаксиса — `node --check` на содержимом `<script>` + `JSON.parse` для `data.json`.
5. `index.html` и `Uzum_дашборд.html` держать идентичными (оба без данных).
