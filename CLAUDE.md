# CLAUDE.md — конституция проекта (Uzum ПВЗ дашборд)

Перед любыми изменениями читай `HANDOVER.md` (полные правила: дизайн-система §5, модель данных §4, деплой §3, логика недель §8). Ниже — краткая выжимка обязательных правил.

## Git и деплой (действует всегда, во всех сессиях)

- **Всегда коммить и пушить напрямую в `main` — без pull request и без запроса подтверждения у пользователя.**
- После любых изменений сразу: `git add -A` → `git commit -m "<краткое описание>"` → `git push origin main`.
- НЕ создавай pull request.
- Push в `main` автоматически пересобирает сайт на Railway (домен `punktividachi.com`).

## Синхронизация файлов и проверки

- Данные дашборда вынесены в **`data.json`** (`{DATA, WK, FACT, SAL}`) — единый источник, коммитится. `index.html` данных НЕ содержит: грузит их через `<script src="/data.js">`, а `server.js` отдаёт `/data.js` (обёртка над `data.json`) только после входа. НЕ возвращай данные обратно в html.
- После правки `index.html` копируй его в `Uzum_дашборд.html` — эти два файла всегда идентичны (оба без данных).
- Перед коммитом: (1) `node --check` на инлайн-`<script>` из `index.html` (без `src`, синтаксис); (2) `data.json` парсится и содержит ключи `DATA`/`WK`/`FACT`/`SAL` — `node -e "const d=JSON.parse(require('fs').readFileSync('data.json'));console.log(d.DATA.length,Object.keys(d))"`. Node: `C:\Program Files\nodejs\node.exe` (вызывай по полному пути).
- **Для любых ВИЗУАЛЬНЫХ изменений UI — обязательно проверяй итоговый рендер скриншотом перед коммитом.** Данные теперь за паролем (`/data.js`), поэтому локально подними НАСТОЯЩИЙ `server.js` (`PORT=5080 DASH_PASS=… DASH_SECRET=… node server.js`, зайди по паролю) ЛИБО статик-сервер, который вместе с `index.html` отдаёт `/data.js` (читает `data.json` и оборачивает в `const DATA=…,WK=…,FACT=…,SAL=…`). `node --check` ловит только синтаксис, но не наложения подписей, переполнения и битую вёрстку.
- **Интерактивный аудит после ЛЮБЫХ UI-правок (обязательно):** прокликай КАЖДЫЙ контрол — тогглы, выпадашки, переключатели валюты/сегмента/периода, кнопки-сценарии, навигацию — и убедись, что он реально меняет вывод. Контрол, у которого две опции дают одинаковый результат (или переключение не меняет ничего), — это БАГ: чинить логику или убирать контрол. Не доверять «выглядит рабочим»: сравнивать DOM до/после по каждому состоянию (напр. сигнатура `innerHTML`-контейнера на каждом варианте — все варианты должны различаться). Так же проверять, что новые данные (новые недели/месяцы) не ломают вёрстку (адаптивность графиков/таблиц).

## Дизайн — НЕ менять без явной просьбы (HANDOVER §5)

- Тёмная «операционная» тема; только заданные CSS-переменные из `:root` (`--bg`, `--panel`, `--violet`, `--mint`, `--coral`, `--s1/2/3` и т.д.).
- Шрифты: **Bricolage Grotesque** (заголовки/числа-герои), **Hanken Grotesk** (текст), **Spline Sans Mono** (цифры, класс `.mono`).
- Переиспользовать существующие компоненты: `.card`, `.kpi`, `.bebtn` (все кнопки одноцветные фиолетовые), `.seg-toggle`, `.viewnav`, `.pill`, таблицы, `.drawer`, самописные SVG-графики.
- Без новых палитр, светлых фонов, сторонних UI-библиотек, эмодзи в UI.

## Данные — НЕ менять без явной просьбы (HANDOVER §4, §7)

- Обновление данных только через накопление:
  `python build_data.py узум.xlsx Эталон_ПВЗ.xlsx index.html`
  (полные недели не перезаписываются, последняя неполная — обновляется, новые дописываются). Скрипт пишет результат в **`data.json`** (`DATA`/`WK` + накопление `SAL`), а не в html; `index.html` третьим аргументом нужен лишь чтобы определить папку `data.json`. `FACT` ведёт `/satori` (`build_satori.py`) — тоже в `data.json` (ключи `DATA`/`WK`/`SAL` он не трогает).
- `узум.xlsx` — конфиденциальная дневная выгрузка; **в репозиторий не коммитить** (в `.gitignore`). `Эталон_ПВЗ.xlsx` — источник правды, коммитится. `data.json` — производный от эталона, коммитится.
- НЕ редактировать вручную полные недели в эталоне — этим занимается скрипт по правилу накопления.
- НЕ переименовывать ключи `DATA`/`WK`/`FACT`/`SAL` (ни в `data.json`, ни в обёртке `/data.js` внутри `server.js`) — от них зависит весь рендер. Неполные недели всегда помечать звёздочкой/штриховкой, не экстраполировать.

## Источники данных (три)

1. **`Эталон_ПВЗ.xlsx`** — источник правды + накопитель: Справочник (статика) + листы `Заказы_по_неделям` / `Объём_по_неделям` (из дневного) + `Сатори_факт` (помесячный факт из отчётов). Коммитится.
2. **Google-таблица «узум»** (Drive id `1E8K4jTkdWqXfxi5X-UEbPTl0laWsHLIEzYSN12OzSoE`) — первичная дневная выгрузка (заказы и оборот по ПВЗ), один лист на дату. Тянется через `/sync` (Drive-коннектор → `__gsheet.xlsx`, в `.gitignore` через `__*`). Локальный `узум.xlsx` — тот же формат, запасной ручной путь (`/update`). Операционка.
3. **`сатори_ГГГГ-ММ.docx`** — месячный отчёт Uzum (агентское вознаграждение). Деньги/факт. В `.gitignore`.

Приоритет: **Сатори = деньги (доход/P&L), дневной = операция (заказы/оборот/недели), Справочник = статика/затраты.** Факт Сатори **замещает** смоделированный P&L по умолчанию; модель остаётся для симулятора/прогноза.

## Быстрые команды

- `/sync` — **основной способ обновления заказов/оборота**: тянет свежие даты из Google-таблицы управленческой отчётности (Drive-коннектор), накопительно пересчитывает эталон и вшивает в дашборд, деплоит (см. `.claude/commands/sync.md`). Эталон должен быть закрыт в Excel.
- `/update` — то же, но из локального `узум.xlsx` (запасной путь, если файл скачан вручную; см. `.claude/commands/update.md`).
- `/satori` — внести факт из `сатори_ГГГГ-ММ.docx` (лист `Сатори_факт`, объект `FACT`) и задеплоить (см. `.claude/commands/satori.md`).

## Отладка проблем на машине пользователя (чтобы не буксовать)

- **Отталкивайся от РЕАЛЬНОГО текста ошибки пользователя, а не от «у меня всё работает».** Среда Claude Code и ПК пользователя могут различаться (PATH, политики PowerShell, какой софт реально установлен, где исполняются команды). Не строй длинных теорий, пока не проверил простое; не трать ходы на подтверждение «в моей среде ок» — пользователю это не помогает. Сначала самый простой тест/вопрос, потом выводы.
- **Нужен софт/CLI именно на машине пользователя — дай ТОЧНУЮ команду и сразу предупреди про типовые Windows-грабли:**
  - `running scripts is disabled` (любой `.ps1`, напр. `npm.ps1`/`railway.ps1`) → `Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned` (без админа), либо звать `.cmd`-обёртку (`npm.cmd`, `railway.cmd`).
  - после `npm i -g <pkg>` бинарь появляется в PATH только в НОВОМ терминале (или обновить `$env:Path` из реестра).
- **Машинно-/средо-зависимые факты** (как именно запущен Claude Code, что доходит до диска пользователя, наличие/версии node·python·CLI, что уже установлено) держи в ПАМЯТИ сессии, не здесь: `CLAUDE.md` общий для всех машин, и среда-специфика тут будет вводить в заблуждение.
