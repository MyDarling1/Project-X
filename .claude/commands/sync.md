«Кнопка» обновления: подтянуть свежие данные из официального API мини-аппа Uzum (`@UzumFranchiseBot`), накопительно пересобрать эталон, обновить `DATA`/`WK`/`COMM` в дашборде, задеплоить. Только данные — дизайн/логику НЕ трогать. **FACT (деньги) этого НЕ касается** — он на `/satori`. (Запасной путь из Google-таблицы — `/sync-gsheet`, только если бот недоступен.)

Этот источник — авторитетный (исправляет занижения Google-таблицы). Доступ — с этой машины (`DESKTOP-2M3J6J7`): API `.uz` иногда не резолвится локально (флап Mullvad), поэтому фетч идёт через DoH-обёртку. Telethon-сессия закеширована в `%TEMP%` → логин обычно не нужен. Подробности: память `[[uzum-franchise-api]]`.

**⚠️ Yandex-щит (anti-DDoS) перед сервером Uzum** челенджит автоматические запросы капчей (`302→showcaptcha`). Обходим НЕ решением капчи, а **клиренс-кукой из реального webview Telegram Desktop** (мини-апп уже прошёл щит как человек): `__uzum_cookies.py` расшифровывает куки `_yasc`/`refresh_token` из `tdata/user_data*/wvbots/EBWebView` (WebView2: AES-GCM + DPAPI-ключ, срезать 32 байта SHA256-домена в начале) → пишет `__uzum_dump/_spravka.txt`. `__uzum_fetch_doh.py` это делает АВТОМАТически перед фетчем. **Требование: Telegram Desktop открыт и мини-апп UzumFranchise хоть раз открыт** (тогда в webview свежие куки). Если фетч пишет «БЛОКИРОВКА капчи» / «give up (captcha?)» — открой мини-апп в Telegram Desktop и повтори `/sync`. `api()` ещё и ретраит капчу повызовно (щит срабатывает не на все вызовы).

**ТОЛЬКО ЗАВЕРШЁННЫЕ СУТКИ (двойной фильтр):** в расчёт идут лишь полностью закрытые дни — для заказов, оборота И комиссии. `__uzum_build.py`/`__uzum_comm.py` отбрасывают день, если: (1) `day >= TODAY_UZ` (текущий день по UTC+5, `datetime.now(utc)+5ч`); ИЛИ (2) он в хвосте и бот его ещё НЕ финализировал — `trailing_incomplete()` (общий хелпер `__uzum_util.py`) идёт от последнего дня назад и выкидывает, пока день неполный (отчиталось точек < максимума ИЛИ сетевых заказов < 45% медианы), стоп на первом полном. Пункт (2) ВАЖЕН: бот ОТСТАЁТ от наших часов, и его последний день в выгрузке бывает частичным (поймали 2026-06-26: 89 заказов / 16 из 18 точек → иначе занизило бы неделю на −15%). Часам не доверяем — завершённость берём из самих данных. Без хардкода даты.

**ИНКРЕМЕНТАЛЬНО (не полная пересборка):** фетч тянет ТОЛЬКО текущий + прошлый месяц (`__uzum_fetch.py` `START_YM`=прош. месяц по UTC+5) — это покрывает текущую+прошлую неделю. Старые недели берутся из НАКОПЛЕННОГО Эталона: `__uzum_build.py`/`__uzum_deploy.py` БОЛЬШЕ НЕ чистят недельные листы → `build_data` накапливает (старые полные недели заморожены, обновляется только хвостовая/добавляются новые). `__uzum_comm.py` СЛИВАЕТ свежие месяцы в существующий `COMM` (старые месяцы, напр. апрель, сохраняются). ⚠️ Компромисс: старые полные недели НЕ авто-корректируются, если Uzum задним числом правит прошлые дни; и синкать надо не реже ~раза в 1.5 мес (окно фетча — 2 мес). Нужна полная пересборка с нуля — верни очистку листов или используй более широкое `START_YM`.

ПОДХОД — ОДИН ПРОХОД (без гейта): фетч → проверка → дифф → СРАЗУ деплой → пуш → отчёт. НЕ жди «go». Дифф — это sanity-check и материал для отчёта, НЕ блокирующее подтверждение. Единственное исключение — **anomaly-abort**: если дифф выглядит сломанным (точка обнулилась, сеть −20%+, исчезли недели, broken-точки в фетче) — СТОП, не деплой, покажи проблему пользователю (обычно недокачанный фетч).

1. **Фетч из бота:**
   ```
   python __uzum_fetch_doh.py
   ```
   Тянет 18 ПВЗ × тек.+прош. месяц в `__uzum_dump/stats_all.json` (DoH-фолбэк на случай флапа `.uz`).
   - Если ошибка авторизации / `no tgWebAppData` / просит код — **сессия протухла**. Сам код/пароль НЕ вводи: попроси пользователя один раз залогинить Telethon (ввести код из Telegram), затем повтори. Это ЕДИНСТВЕННЫЙ шаг, где нужен пользователь.

2. **Проверь, что фетч чистый** (все 18 точек, без пустых/`error`-месяцев):
   ```
   python -c "import json;b=json.load(open('__uzum_dump/stats_all.json',encoding='utf-8'));br=[p['name'] for p in b['points'] if sum(len(m.get('days',[])) for m in p['months'].values())==0];print('points',len(b['points']),'broken',br or 'NONE')"
   ```
   Если есть broken — перезапусти фетч (флап DNS); не строй на «отравленных» данных.

3. **Кандидат + ДИФФ (live не трогаем):**
   ```
   python __uzum_build.py
   ```
   Пишет `__data_candidate.json` и печатает дифф vs текущего `data.json` (недели, сеть по неделям, по-ПВЗ). Прогони как **sanity-check**: если аномалия (точка обнулилась, сеть −20%+, недели исчезли) — СТОП и сообщи пользователю. Если всё нормально — НЕ жди подтверждения, сразу деплой (шаг 4). Дифф покажешь в финальном отчёте.

4. **Деплой в живое** (сразу после чистого диффа, без подтверждения):
   ```
   python __uzum_deploy.py
   ```
   Делает бэкап (`__data_backup.json`/`__etalon_backup.xlsx`), прогоняет `build_data.py` БЕЗ очистки листов (накопление: старые недели заморожены, FACT/SAL сохраняются), сливает блок `COMM` (`__uzum_comm.py` — комиссия/самоокупаемость, старые месяцы целы), гоняет проверки целостности (18 ПВЗ, FACT/SAL на месте, == кандидату). Если упал — откат из `__*_backup`, разберись.

5. **Синхронизируй копию:** `index.html` → `Uzum_дашборд.html` тронуты НЕ были (меняются только данные), но сверь хэш на всякий: `cmp -s index.html Uzum_дашборд.html`.
